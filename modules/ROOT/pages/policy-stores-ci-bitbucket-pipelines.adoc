= Cerbos Hub Bitbucket Pipelines Integration

Automate policy uploads from Bitbucket by running `cerbosctl` inside Bitbucket Pipelines. The pipeline uses Bitbucket's managed Docker service so you can keep your repository free of extra tooling.

== Prerequisites

* A Bitbucket Cloud repository with Pipelines enabled (`Repository settings → Pipelines → Settings`).
* A Cerbos Hub policy store and its Store ID.
* Cerbos Hub API credentials (`CERBOS_HUB_CLIENT_ID` and `CERBOS_HUB_CLIENT_SECRET`).

== Create `bitbucket-pipelines.yml`

. In the root of your repository, add a file named `bitbucket-pipelines.yml` with the configuration below:

[source,yaml]
----
# bitbucket-pipelines.yml
pipelines:
  branches:
    main:
      - step:
          name: Upload policies to Cerbos Hub
          services:
            - docker
          script:
            - |
              docker run --rm \
                -e CERBOS_HUB_CLIENT_ID \
                -e CERBOS_HUB_CLIENT_SECRET \
                -v "$BITBUCKET_CLONE_DIR":/app \
                ghcr.io/cerbos/cerbosctl:latest \
                hub store replace-files \
                  --store-id="<STORE_ID>" \
                  --message="Policy upload from Bitbucket" \
                  /app
                  # Optional: append --api-endpoint="https://<region>.cerbos.cloud" if Cerbos provided a custom endpoint
----

Update `<STORE_ID>` with the identifier from Cerbos Hub. Adjust the branch filter or pipeline name if needed.

== Store Cerbos Hub credentials

. Open *Repository settings* in Bitbucket and navigate to *Pipelines → Repository variables*.
. Add a variable named `CERBOS_HUB_CLIENT_ID`, paste your client ID value, and select *Secured* before saving.
. Add another secured variable named `CERBOS_HUB_CLIENT_SECRET` with the matching secret value.

== Commit and monitor the pipeline

. Commit `bitbucket-pipelines.yml` and push it to the `main` branch.
. Open the *Pipelines* tab in Bitbucket to observe the new run. Selecting the step reveals the Docker log generated by `cerbosctl` so you can confirm that the policy store was updated.
. Future pushes to the tracked branch will re-run the same step and keep your policy store synchronized.
